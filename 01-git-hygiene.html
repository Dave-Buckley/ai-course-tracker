Git Hygiene

8 focus blocks ~4 hours total Phase 2 prep

What is Git, in one sentence? Git is a tool that keeps a saved history of every change you make to your code, on your computer. GitHub is a website that hosts copies of those histories online so you can share them, back them up, and collaborate. You'll use both together for the next 18 months.

Every later worksheet ends with "commit and push to GitHub". Spending 4 hours getting the Git muscle warm now means zero friction for the next 16 worksheets. By the end of this one you'll have made real branches, opened a real Pull Request, and know how to undo mistakes without losing work.

Setup + identity

Tell Git who you are, then make your first practice repo on GitHub. By the end of this block your name shows up on a commit you pushed online.

1. Confirm Git is installed

What: check that Git exists on your computer.

Why: if it's missing, every other step fails.

In VS Code's terminal (View → Terminal), run:

git --version
Worked? You see something like git version 2.53.0.windows.1. If you see "command not found", install Git from git-scm.com and restart PowerShell.

2. Set your global Git identity

What: tell Git the name and email to attach to every commit you make.

Why: GitHub uses the email to link commits back to your account so they show up on your profile. "Global" means this applies to every Git project on this computer, not just one.

Run these two lines (replace your-github-email with the address from your GitHub email settings — see note below):

git config --global user.name "David Buckley"
git config --global user.email "your-github-email@example.com"
Which email? Open github.com/settings/emails. If "Keep my email addresses private" is ticked, GitHub shows a no-reply email like 12345678+Dave-Buckley@users.noreply.github.com — use that. Otherwise use your primary email shown on the page.
Worked? Run git config --get user.name — it prints "David Buckley". Then git config --get user.email — it prints the email you set.

3. Set the default branch name and editor

What: tell Git that new repos start with a branch called main, and to use VS Code when it needs to ask you for a longer message.

Why: main is the modern industry default (older Git used master). VS Code is friendlier than the default editor for typing commit messages.

Run these two lines:

git config --global init.defaultBranch main
git config --global core.editor "code --wait"
Worked? No output means success. git config --get init.defaultBranch prints main.

Conventional Commits

Practice writing commit messages in the format the industry uses. Every commit message starts with a single-word prefix saying what type of change it is.

Why this matters Hiring managers skim your git log to gauge how disciplined you are. A log full of "stuff", "fix", "more changes" reads as careless. Prefixed messages (feat:, fix:, chore:) look professional and let tools auto-generate changelogs. The format is from conventionalcommits.org.

1. Read the spec (5 min)

What: open the Conventional Commits website and skim the "Summary" section.

Why: the rules are simple, you'll absorb them in one pass.

Open conventionalcommits.org. Read until "Examples". Stop there.

Worked? You can name what each prefix means: feat, fix, chore, docs, refactor.

Branching basics

What's a branch? A branch is a parallel copy of your code where you can experiment without affecting the main version. Think of it like making a new save file in a video game so you can try a risky move — if it doesn't work, you can go back to the original save. main is the always-working version. feat/about-page is where you'd build a new "about" page without breaking anything yet.

1. Make a new branch in PowerShell

Make sure you're inside 01-git-hygiene, then:

git checkout -b feat/about-page

Plain English: "create a new branch called feat/about-page and switch to it." The -b means "create".

Worked? You see Switched to a new branch 'feat/about-page'. git branch shows two branches; the asterisk is on feat/about-page.

2. Create about.md on this branch

In VS Code:

  1. Hover over 01-GIT-HYGIENE, click "New File".
  2. Name: about.md
  3. Type: 
    # About

This page describes who I am.
  4. Save with Ctrl+S.

File location: C:\Users\David\Documents\AI Course\01-git-hygiene\about.md

In VS Code's terminal (View → Terminal):

git add about.md
git commit -m "feat: add about page draft"
Worked? git log --oneline on this branch shows your new commit at the top.

3. Watch the file disappear and reappear as you switch branches

Why this matters: proves that branches are isolated. about.md only exists on feat/about-page — switching to main hides it.

In VS Code's terminal (View → Terminal):

git switch main

Look at the file panel in VS Code: about.md is gone.

git switch feat/about-page

It's back. Switch back to main for the next block:

git switch main
Worked? You watched about.md disappear and reappear in VS Code's file panel as you switched branches.

3. List branches and delete an unused one

You see all your branches; the current one has an asterisk.

If you want to delete feat/about-page (don't yet — keep it for block 5), the command would be: git branch -d feat/about-page. Git will refuse if the branch has unmerged work.

Worked? You can list branches and you understand that -d is the delete flag.

Merging and rebasing — what they really do

Two ways to combine work from one branch into another. Both achieve the same final code, but produce different-looking history.

1. Try a merge

First, make a commit on main so the two branches diverge:

In VS Code, create a new file:

  2. Name: status.txt
  3. Type one line: updated
  4. Save.

File location: C:\Users\David\Documents\AI Course\01-git-hygiene\status.txt

In VS Code's terminal (View → Terminal), commit it on main, then switch to the feature branch and merge:

git add status.txt
git commit -m "chore: add status file"
git switch feat/about-page

Git creates a "merge commit" tying both lines of work together.

Worked? git log --graph --oneline --all shows two lines that come together at a merge commit.

2. Try a rebase on a fresh branch

Switch back to main, make a new branch:

git switch main
git checkout -b feat/contact

In VS Code, create a new file:

  1. Hover over 01-GIT-HYGIENE, click "New File".
  2. Name: contact.md
  3. Type one line: Contact info goes here.
  4. Save.

Rule of thumb Use merge on shared branches (work other people pull from). Use rebase on your own local branches before opening a PR, to keep history clean. Never rebase a branch you've already pushed to GitHub if anyone else might have pulled it.

Pull Requests end-to-end

What's a Pull Request? A PR is a GitHub feature where you propose merging a branch into main via a webpage, optionally getting feedback first. Even working alone, doing changes via PRs creates a clean review trail that hiring managers love. The flow is: branch → commits → push → open PR on github.com → review → squash-merge → delete branch.

1. Switch to main + make a new feature branch

In VS Code's terminal (View → Terminal):

git switch main
git pull
git checkout -b feat/footer

git pull downloads any new commits from GitHub before you start. checkout -b creates and switches to the new branch.

2. Create footer.md on the branch

In VS Code:

  1. Hover over 01-GIT-HYGIENE, click "New File".
  2. Name: footer.md
  3. Type: 
    # Footer

© 2026 David Buckley.
  4. Save.

File location: C:\Users\David\Documents\AI Course\01-git-hygiene\footer.md

In VS Code's terminal (View → Terminal):

git add footer.md
git commit -m "feat: add footer"
git push -u origin feat/footer

The -u means "remember this branch links to origin/feat/footer" so future git push calls know where to send. You only need -u the first time.

Worked? The push output ends with a URL like https://github.com/Dave-Buckley/git-practice/pull/new/feat/footer. Copy it.

2. Open a PR on GitHub.com

Open the URL from step 1 in Chrome (or visit github.com/Dave-Buckley/git-practice/pulls and click "New pull request"). Fill in:

  • Title: "Add footer" (the conventional-commit message will be the title by default)
  • Description: 
    ## Summary
- Adds footer.md with placeholder text

## Test plan
- [ ] File renders on the page

Click Create pull request.

Worked? You're now on a page showing the PR with your changes.

3. Squash and merge

What: "Squash and merge" combines all the commits on your branch into ONE commit on main.

Why: keeps main's history clean — one PR = one entry in the log. The full commit-by-commit history is still preserved on the (now-closed) branch if you ever need it.

Click the green Squash and merge button. Confirm. Then click Delete branch when offered.

Worked? The PR shows "Merged" with a purple badge.

4. Pull the merge back to your local main

Back in PowerShell:

git switch main
git pull
git branch -d feat/footer

"Pull" downloads the new commit on main from GitHub. Then you delete the local copy of the branch since it's already merged.

Worked? git log --oneline on main shows the new "Add footer" commit at the top.
Block 6 ~30 min ▶ Open VS Code here

Recovery: undoing things safely

Why this block exists The single biggest source of "I lost my work" panic is misusing reset/revert/stash. Once you've used each one once, you'll never lose work to Git again.

1. reset --soft — undo a commit but keep the changes

What: erase the most recent commit but leave the file changes ready to be re-committed.

First, make a throwaway commit. In VS Code, create a new file:

  1. Hover over 01-GIT-HYGIENE, click "New File".
  2. Name: undo-me.txt
  3. Type one line: throwaway content
  4. Save.

File location: C:\Users\David\Documents\AI Course\01-git-hygiene\undo-me.txt

In VS Code's terminal (View → Terminal):

git add undo-me.txt
git commit -m "chore: throwaway commit"
git reset --soft HEAD~1

HEAD~1 means "the commit one before the current one". reset --soft moves the branch pointer back but leaves your file changes staged.

Worked? git status shows undo-me.txt as a staged change again. The commit is gone but the file isn't.

2. reset --hard — DESTROY changes (local-only branches!)

Danger --hard permanently throws away changes. Only use on branches you haven't pushed yet. Never use on main if you've ever shared it.

Recommit, then nuke it:

git commit -m "chore: about to be destroyed"
git reset --hard HEAD~1
Worked? The file undo-me.txt is gone from your folder. git status shows nothing to commit.

3. revert — the SAFE way to undo a published commit

What: creates a NEW commit that undoes a previous one, without rewriting history.

Why: if you've already pushed a bad commit to GitHub, this is how you remove its effect without messing up anyone else who pulled it.

Find a commit hash from git log, then:

git log --oneline -5
git revert <paste-a-hash>

A commit message editor opens — accept the default by closing it.

Worked? A new commit appears at the top of git log with "Revert" in the message.

4. stash — park changes temporarily

What: shelve uncommitted changes so you can switch branches without committing or losing them.

In VS Code, create a new file:

  1. Hover over 01-GIT-HYGIENE, click "New File".
  2. Name: wip.txt
  3. Type one line: work in progress
  4. Save.

File location: C:\Users\David\Documents\AI Course\01-git-hygiene\wip.txt

Now stash it, switch branch, come back, unstash:

git add wip.txt
git stash
git switch main

Look at the file panel — wip.txt is gone (parked in the stash).

git switch -
git stash pop

git switch - means "switch back to the previous branch". stash pop restores your stashed changes.

Worked? wip.txt reappears in the file panel after stash pop.

.gitignore, log, blame, diff

1. Expand your .gitignore

What: the file you created in Block 2 only has *.log. Real projects ignore more — secrets, dependency folders, OS junk. Update it now.

Why: some files (passwords in .env, the huge node_modules/ folder, OS metadata like .DS_Store) should never go into Git. .gitignore is the gatekeeper.

In VS Code, open .gitignore from the file panel and replace its contents with:

.env
.venv/
node_modules/
*.log
.DS_Store

File location: C:\Users\David\Documents\AI Course\01-git-hygiene\.gitignore

Save (Ctrl+S), then commit:

git add .gitignore
git commit -m "chore: expand gitignore"

Now test it. In VS Code, create a new file called .env with any text inside, save it.

Worked? Run git status — .env doesn't appear as untracked. Git is ignoring it. (Don't actually commit secrets.)

2. Read history with log

Try these:

git log --oneline -10
git log --graph --oneline --all
git log -p <file>        # shows every change to that file
Worked? You can see history at different levels of detail.

3. Find who/when with blame

What: shows the last commit that touched each line of a file.

Run on your README:

git blame README.md
Worked? Each line is prefixed with a commit hash, your name, and a date.

4. See what changed with diff

Edit any file (don't commit yet), then:

git diff                # changes you haven't staged
git add <file>
git diff --staged       # changes that ARE staged but not committed
git diff main..HEAD     # changes between main and current branch
Worked? Each command shows different layers of pending changes.
