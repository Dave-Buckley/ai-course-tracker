# DNS notes

- DNS converts a domain (example.com) into an IP (93.184.216.34).
- Record types: A (IPv4), AAAA (IPv6), CNAME (alias), MX (mail).
- Caching exists at every level (browser, OS, ISP) — that's why DNS feels instant.
- TTL = how long a record can be cached before re-checking.

TCP basics

sequenceDiagram participant C as Client participant S as Server C->>S: SYN S-->>C: SYN-ACK C->>S: ACK Note over C,S: Connection established

Read + notes

Open MDN's TCP overview (3 min). Create 02-tcp.md:

# TCP notes

- 3-way handshake (SYN, SYN-ACK, ACK) establishes a reliable connection.
- TCP guarantees ordering and delivery (retransmits lost packets). UDP doesn't.
- Port = 16-bit number identifying the application: 80 = HTTP, 443 = HTTPS, 22 = SSH.

HTTP requests with curl

1. Verbose GET

curl -v https://example.com

Request line: GET / HTTP/2. Then headers. Then a blank line. Then the response body.

2. POST with a JSON body

curl -X POST -H "Content-Type: application/json" -d '{"foo":"bar"}' https://httpbin.org/post

3. PUT, DELETE, HEAD

curl -X PUT -d 'hello' https://httpbin.org/put
curl -X DELETE https://httpbin.org/delete
curl -I https://example.com
Worked? Each command shows different request methods + responses. -I shows headers only.

Headers + cookies

1. Headers reference — 05-headers.md

Create. Type:

# Common headers

Request:
- Accept              — what content types we want back
- Authorization       — credentials (Bearer token, Basic auth)
- User-Agent          — what client we are
- Cookie              — sends session info to server

Response:
- Content-Type        — what type of body we're sending
- Cache-Control       — how long this response can be cached
- Set-Cookie          — server sets a cookie on the client

2. Send + receive cookies

curl -c cookies.txt -b cookies.txt "https://httpbin.org/cookies/set?session=abc123"
curl -b cookies.txt https://httpbin.org/cookies

Open cookies.txt in VS Code — the cookie was saved.

Worked? Second request shows session=abc123 in the response.

2. Notes — 06-https.md

Create. Type:

# HTTPS / TLS notes

- TLS provides: encryption (privacy), integrity (no tampering), authentication (server is who it claims).
- A certificate is a signed statement: "this public key belongs to example.com".
- A Certificate Authority (CA) signs certs. Browsers ship trusting ~150 CAs.
- Self-signed cert = signed by its own private key (no CA). Browsers warn because there's no chain of trust.
- Mixed content = HTTPS page loading HTTP resources. Browsers block by default.

REST conventions

flowchart LR A[Resource: users] --> B["GET /users → list"] A --> C["POST /users → create"] A --> D["GET /users/42 → read one"] A --> E["PATCH /users/42 → update"] A --> F["DELETE /users/42 → delete"]

1. REST notes — 07-rest.md

Create. Type:

# REST principles

- Resources have URLs (nouns: /users, not /getUsers).
- HTTP methods map to CRUD: GET=read, POST=create, PUT/PATCH=update, DELETE=delete.
- Stateless — each request carries everything needed (auth tokens, etc.).
- Idempotent methods (GET, PUT, DELETE) can be safely retried.
- JSON is the de-facto media type.
