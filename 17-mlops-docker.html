First Dockerfile

1. Create a tiny FastAPI app

In 17-mlops-docker, create folder 01-hello/ and inside:

01-hello/app.py:

from fastapi import FastAPI
app = FastAPI()

@app.get("/")
def root(): return {"hello": "from docker"}

01-hello/requirements.txt:

fastapi
uvicorn[standard]

01-hello/Dockerfile:

FROM python:3.13-slim
WORKDIR /app
COPY requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt
COPY app.py .
CMD ["uvicorn", "app:app", "--host", "0.0.0.0", "--port", "8000"]

2. Build + run

cd 01-hello
docker build -t hello-fastapi .
docker run -p 8000:8000 hello-fastapi

Open localhost:8000 in Chrome.

Worked? Browser shows {"hello": "from docker"}. Stop with Ctrl+C.

Multi-stage builds + .dockerignore

1. Add .dockerignore

01-hello/.dockerignore:

.git
.venv
__pycache__
*.log
.DS_Store
.env

Why: these files don't belong in your image. Smaller image, less risk of leaking secrets.

2. Multi-stage Dockerfile

Replace Dockerfile with:

FROM python:3.13-slim AS builder
WORKDIR /app
COPY requirements.txt .
RUN pip install --no-cache-dir --target=/install -r requirements.txt

FROM python:3.13-slim
WORKDIR /app
COPY --from=builder /install /usr/local/lib/python3.13/site-packages
COPY app.py .
CMD ["uvicorn", "app:app", "--host", "0.0.0.0", "--port", "8000"]

Rebuild + compare sizes:

docker build -t hello-fastapi:multi .
docker images | findstr hello-fastapi
Worked? Both versions build and run identically. Multi-stage may be a bit smaller.

Env vars + volumes

1. Read env in the app

Update app.py:

from fastapi import FastAPI
import os

app = FastAPI()
MESSAGE = os.getenv("MESSAGE", "default-hello")

@app.get("/")
def root(): return {"message": MESSAGE}

Rebuild + run with env override:

docker build -t hello-env .
docker run -e MESSAGE="hi from env" -p 8000:8000 hello-env
Worked? Browser shows "hi from env".

2. Try a volume mount

Stop the container. In a new terminal in 01-hello/:

mkdir data
docker run -v "${PWD}/data:/app/data" -p 8000:8000 hello-env

Files in your local data/ are shared with /app/data inside the container.

docker-compose for multi-service

Create 02-compose/

Copy app.py, requirements.txt, Dockerfile from 01-hello/ into 02-compose/.

Create 02-compose/docker-compose.yml:

services:
  app:
    build: .
    ports:
      - "8000:8000"
    environment:
      DATABASE_URL: postgresql://pg:pass@db:5432/app
    depends_on:
      - db
  db:
    image: postgres:16
    environment:
      POSTGRES_USER: pg
      POSTGRES_PASSWORD: pass
      POSTGRES_DB: app
    volumes:
      - pgdata:/var/lib/postgresql/data
volumes:
  pgdata:

Run:

cd 02-compose
docker compose up
Worked? Both containers start. App on localhost:8000. Stop with Ctrl+C, then docker compose down to clean up.

Deploy to Fly.io

1. Install flyctl + sign up

In PowerShell:

iwr https://fly.io/install.ps1 -useb | iex

Restart the terminal. Then:

fly auth signup

Sign up via the browser window that opens (free tier, no card needed).

2. Deploy

cd 01-hello
fly launch --no-deploy
fly deploy

Accept the defaults during fly launch — it generates a fly.toml.

Worked? fly status shows the app running. Visit the URL it prints.

3. Tail logs

fly logs

Hit your Fly URL in Chrome — see the request appear in the log stream.

CI with GitHub Actions

1. Create the workflow file

In your 01-hello/ folder, create .github/workflows/ci.yml (yes, two nested folders starting with a dot):

name: ci
on: [push, pull_request]
jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-python@v5
        with:
          python-version: "3.13"
      - run: pip install -r requirements.txt pytest httpx
      - run: pytest -v

2. Add a tiny test

Create tests/test_app.py:

from fastapi.testclient import TestClient
from app import app

def test_root():
    client = TestClient(app)
    r = client.get("/")
    assert r.status_code == 200

3. Push + watch CI run

git init
git add .
git commit -m "feat(docker): hello-fastapi with CI"
gh repo create hello-fastapi --public --source=. --remote=origin --push

Open the repo on GitHub → Actions tab. Watch the green check.

Worked? CI passes. Future pushes auto-test.

Containerise + deploy your RAG app

Take Worksheet 15's RAG app. Containerise. Deploy. CI it.

1. Add a Dockerfile to 15-rag/

FROM python:3.13-slim
WORKDIR /app
COPY requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt
COPY . .
CMD ["uvicorn", "app.main:app", "--host", "0.0.0.0", "--port", "8000"]

Add .dockerignore with .venv, __pycache__, .env, chroma_db.

Generate requirements.txt from the venv: pip freeze > requirements.txt.

2. Build + test locally

docker build -t rag-demo .
docker run -p 8000:8000 -e GEMINI_API_KEY=your-key rag-demo

Test the endpoint as in Worksheet 15.

3. Deploy to Fly.io

fly launch --no-deploy
fly secrets set GEMINI_API_KEY=your-key
fly deploy

Visit the public URL. Test from a real device.

4. Add CI to the repo

Same .github/workflows/ci.yml as block 7. Commit + push.

Worked? Public URL serves the RAG API. CI green. Counts towards Phase 5 MKT-01 / DAI-04.
